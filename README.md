# Распознавание диалогов и терминов каталога в speech2text системе

Проект обрабатывает русскоязычные аудиозаписи диалогов и формирует
структурированный результат:

- распознаёт речь с помощью Faster-Whisper;
- отделяет речь от тишины с Silero VAD;
- определяет спикеров через pyannote Community-1;
- ищет названия продуктов из каталога с CTC context graph на NeMo;
- связывает реплики и найденные термины со спикерами;
- при наличии эталонной разметки считает WER, CER, DER и метрики терминологии.

Данный проект выполнен в рамках производственной практики.

## Архитектура

Пайплайн запускается из `src.main`. Он загружает YAML-конфигурацию, выполняет предварительную проверку файлов и ресурсов, после чего последовательно запускает этапы обработки.

```text
Аудиофайл
    |
Preflight-проверка
    |- наличие входного файла;
    |- наличие моделей и каталога;
    |- наличие HF_TOKEN;
    |- доступность CUDA.
    |
Нормализация аудио
    |- mono; 16 kHz; PCM_16 WAV.
    |
Silero VAD
    |- выделение интервалов с речью.
    |
Обработка речевых интервалов
    |--- Faster-Whisper
    │  - распознавание речи;
    │  - формирование ASR-сегментов.
    │
    |--- pyannote Community-1
    │  - определение интервалов спикеров.
    │
    |--- NeMo CTC context graph
    |  - поиск упоминаний терминов каталога.
    |
Сопоставление результатов
    |- назначение спикеров ASR-сегментам;
    |- привязка терминов к репликам;
    |- сопоставление по временному перекрытию.
    |
Формирование результата
    |- JSON, stdout;
    |- опциональный Typst-отчёт.
```

| Файл | Содержание |
| --- | --- |
| `src/main.py` | CLI, загрузка конфигурации, preflight, запись результата |
| `src/app_config.py` | чтение и строгая проверка YAML-конфигурации |
| `src/pipeline.py` | последовательное выполнение этапов и сборка результата |
| `src/audio/normalization.py` | приведение входного аудио к единому формату |
| `src/vad/silero_vad.py` | поиск фрагментов с речью |
| `src/asr/whisper.py` | распознавание речи и временные метки слов |
| `src/diarization/pyannote.py` | определение анонимных спикеров и их реплик |
| `src/asr/terminology.py`, `src/context_graph.py` | акустический поиск терминов каталога без дообучения ASR |
| `src/diarization/transcript.py` | привязка ASR-сегментов и терминов к спикерам |
| `src/evaluation/quality.py` | WER, CER, DER и метрики терминологии по эталону |
| `src/reporting/render.py` | консольный диалог и Typst-отчёт |

## Требования
- Python 3.10;
- `uv`;
- FFmpeg — нужен `pydub` для чтения и нарезки аудио;
- `HF_TOKEN` для
  `pyannote/speaker-diarization-community-1`.

В `pyproject.toml` закреплены CUDA-сборки PyTorch. На машине без NVIDIA GPU
пайплайн может работать на CPU, но установка зависимостей и скорость работы
могут отличаться.

Перед первым запуском примите условия использования модели
`pyannote/speaker-diarization-community-1` на Hugging Face и задайте токен:

```bash
export HF_TOKEN=hf_...
```

## Файлы моделей и каталога

В конфигурации по умолчанию проект ожидает следующую структуру. Имена папок
можно изменить в `config/default.yaml`, указав другие `model_path` и
`catalog_path`.

```text
asr_practice/
├── config/
│   └── default.yaml
├── models/
│   └── stt_ru_conformer_ctc_large.nemo  # локальная NeMo CTC-модель
├── resources/
│   └── fire_catalog.json                # каталог терминов
└── recordings/
    ├── dialogue.wav
    └── dialogue.reference.json          # необязательно, для метрик
```

`models/stt_ru_conformer_ctc_large.nemo` — бинарный файл NeMo-модели. Его
нельзя заменить JSON-файлом или папкой с весами: в `model_path` должен быть
путь именно к существующему файлу `.nemo`.

### Каталог терминов

`resources/fire_catalog.json` — UTF-8 JSON со следующей структурой:

```json
{
  "catalog_version": "1.0",
  "products": [
    {
      "id": "rubezh-2op",
      "canonical_name": "Рубеж-2ОП",
      "spoken_forms": [
        "рубеж два оп",
        "рубеж 2 оп"
      ],
      "ctc_forms": [
        "рубеж два о п"
      ]
    },
    {
      "id": "ip-212-141",
      "canonical_name": "ИП 212-141",
      "spoken_forms": [
        "ип двести двенадцать сто сорок один"
      ]
    }
  ]
}
```

Обязательные поля каждого продукта:

| Поле | Тип | Требование |
| --- | --- | --- |
| `id` | строка | уникальный стабильный идентификатор|
| `canonical_name` | строка | уникальное отображаемое название продукта |
| `spoken_forms` | массив строк | хотя бы один вариант того, как название произносится в аудио |

Необязательное поле `ctc_forms` — массив дополнительных вариантов для
CTC-декодера. Его удобно использовать для раздельного произношения букв,
цифр и других форм, которые модель распознаёт лучше, чем обычное написание.
Все варианты из `spoken_forms` и `ctc_forms` объединяются при построении
графа. Повторять одинаковые строки не нужно.

Поля `catalog_version`, `products`, `id`, `canonical_name` и `spoken_forms`
обязательны. Загрузчик также проверяет, что `id` и `canonical_name` не
повторяются, а `spoken_forms` не пустой.

Папки `models/` и `resources/` намеренно не хранятся в репозитории: они могут
содержать большие модели и данные проекта.

## Быстрый запуск

```bash
uv run python -m src.main path/to/dialogue.wav
```

По умолчанию рядом с аудиофайлом будет создан файл
`dialogue.diarized.json`.

Полезные параметры:

```bash
# Явно задать выходной JSON
uv run python -m src.main input.wav --output artifacts/result.json

# Использовать GPU для ASR и распознавания терминологии
uv run python -m src.main input.wav --device cuda

# Указать известное число участников разговора
uv run python -m src.main input.wav --num-speakers 2

# Создать Typst-отчёт в report.typ
uv run python -m src.main input.wav --typst

# Использовать другую YAML-конфигурацию
uv run python -m src.main input.wav --config config/default.yaml
```

Перед загрузкой моделей программа выполняет preflight-проверку: наличие
аудио, моделей и каталога, `HF_TOKEN`, доступность CUDA и кэш модели pyannote.
Если проверка не пройдена, инференс не запускается.

## Конфигурация

Основная конфигурация находится в [`config/default.yaml`](config/default.yaml).

| Раздел | Назначение |
| --- | --- |
| `asr` | модель Faster-Whisper, устройство, beam search и язык |
| `vad` | параметры Silero VAD; текущий адаптер работает на CPU |
| `diarization` | модель и режим pyannote, ограничения на число спикеров |
| `terminology` | пути к каталогу и NeMo-модели, пороги CTC context graph |
| `report` | цвета и правило объединения соседних реплик спикера |

Значение `device: auto` выбирает CUDA только при наличии GPU с памятью не
менее 8 ГиБ; иначе выбирается CPU. Параметр `--device` переопределяет
устройство для ASR и терминологии. Устройство диаризации задаётся отдельно в
YAML.

## Формат результата

Выходной JSON содержит:

- полный распознанный текст и язык;
- `speaker_turns` — временные интервалы спикеров;
- `segments` — ASR-сегменты с назначенным спикером;
- `product_mentions` — найденные позиции каталога, оценку и спикера;
- `dialogue` — реплики в виде, удобном для просмотра;
- `quality` — метрики качества, если найдена эталонная разметка;
- `runtime` — длительность этапов и фактические устройства;
- `preflight` — результат проверки перед запуском.

Для расчёта метрик рядом с аудио разместите файл
`<имя_аудио>.reference.json`. Например, для `recordings/dialogue.wav` это
`recordings/dialogue.reference.json`:

```json
{
  "reference": "Здравствуйте, нужен Рубеж-2ОП.",
  "num_speakers": 2,
  "expected_product_ids": ["rubezh-2op"],
  "turns": [
    {
      "start": 0.0,
      "end": 2.4,
      "speaker_id": "customer",
      "text": "Здравствуйте, нужен Рубеж-2ОП."
    },
    {
      "start": 2.8,
      "end": 4.1,
      "speaker_id": "operator",
      "text": "Сейчас проверю наличие."
    }
  ]
}
```

`reference` нужен для WER и CER. Если его нет, текст собирается из полей
`turns[].text`. `turns` включает DER, `num_speakers` (или
`expected_num_speakers`) включает проверку числа спикеров, а
`expected_product_ids` — precision, recall и F1 терминологии. Если файла нет,
распознавание всё равно выполняется, а в `quality` будет указано, что метрики
недоступны.

Также с заданным флагом `--typst` будет возвращён Typst-отчёт. Пример содержимого такого отчёта можно увидеть в `src/report.pdf`.

## Ограничения

- Каталожное распознавание требует локальную `.nemo`-модель и каталог с
  произносимыми вариантами продуктов.
- Для лучшей скорости ASR и терминологии рекомендуется CUDA; VAD в текущей
  реализации остаётся на CPU.
- В рамках данной задачи дообучение _не применялось_ в силу отсутствия доменных данных.
